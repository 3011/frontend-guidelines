# 前端实施工作流

## 1. 建立上下文

修改前检查：

- 页面入口和用户主要任务；
- 当前框架与已有组件；
- 同类页面和共享模式；
- 数据状态、权限与错误路径；
- 现有测试和运行方式。

## 2. 定位根因

记录：

- 用户可观察的症状；
- 触发条件；
- 根因所在层级；
- 可能受影响的同类界面；
- 不应被修改的边界。

## 3. 选择修复层级

按以下顺序判断：

1. 数据或状态模型是否错误；
2. 共享交互模式是否错误；
3. 页面组合是否错误；
4. 仅当前实例是否特殊。

不要默认从当前页面添加补丁。

## 4. 实施

- 保持修改集中；
- 复用项目现有语言和视觉体系；
- 不无理由增加依赖；
- 不重写无关区域；
- 对复杂状态转换写清楚时序；
- 保留可回滚性。

## 5. 验证

先运行静态检查和现有测试，再进行真实浏览器验证。验证结果应覆盖规则要求，而不是只证明页面能加载。

涉及动画、定位和焦点的问题，应验证时间序列：开始前、变化中、完成后。

## 6. 同类审计

修复共享根因后，搜索所有使用位置并检查：

- 是否自动得到修复；
- 是否存在依赖旧行为的例外；
- 是否需要迁移；
- 是否需要增加回归测试。

## 7. 交付报告

报告至少包括：

```text
根因：
修改层级：
涉及规则：
验证证据：
未验证内容：
风险与回滚：
```
